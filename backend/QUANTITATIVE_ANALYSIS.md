# Quantitative Process Analysis Module

Implements the Chapter 7 ("Quantitative Process Analysis") toolkit from
Dumas et al., *Fundamentals of Business Process Management* (2nd ed.), over
this repo's existing process graph representation, and wires it into the
existing RL redesign pipeline to produce a before/after report.

- `app/core/quantitative_analysis.py` - the module. Every function is
  generic over the schema (walks whatever `gateways[]`/`process_task[]`
  graph it's given, not hardcoded to the sample process).
- `report.py` - CLI entry point: `python report.py <as-is JSON path>`
  writes `<code>_analysis.json` and `<code>_report.md`.
- `app/tests/test_quantitative_analysis.py` - unit tests.

This document explains the assumptions and formula choices behind the
numbers, since the output JSON deliberately contains none of that (see
"Required JSON output contract" - it's clean results only).

## What already existed vs. what was added

This repo already had a working RL-based redesign engine
(`app/core/parser.py`, `metrics.py`, `heuristics.py`, `trainer.py`,
`replay.py`, `services/redesign_service.py`) that computes cycle time and
cost via a probability-weighted graph walk and generates a to-be process by
searching for a good heuristic sequence (Q-learning) - it does *not* take a
hand-authored to-be file. `compare()` reuses that pipeline rather than
duplicating it, and adds the parts Chapter 7 asks for that weren't there
yet: the Repetition/loop pattern, Cycle Time Efficiency, dual-currency cost
tracking, and resource utilization / Little's Law.

## `POST /redesign` now returns this suite too, via one shared code path

`app/services/redesign_service.py::redesign()` (the live endpoint) and
`compare()` (the CLI/thesis-report path) both call
`analyze_before_after(baseline_graph, final_graph)`, one function that
runs `_full_analysis()` (flow analysis + CTE + resource utilization +
Little's Law) on both graphs and computes the improvement deltas. Neither
caller re-derives these numbers independently - this was a deliberate fix
for a duplication problem an earlier audit pass flagged: this project used
to have two Python implementations of the before/after-assembly logic (one
inline in `redesign_service.redesign()`, one inline in `compare()`) that
could silently drift apart. There's now one.

The two callers still differ in exactly one place, by design:
`redesign_service.redesign()` renames `cost_currency_normalized` to
`cost` for its own `before`/`after` (preserving this endpoint's
already-correct, already-shipped currency-normalized meaning of "cost" -
see `KNOWN_LIMITATIONS.md`), and recomputes `improvement.cost_percent`
from that basis. `compare()` keeps `cost` as the raw, mixed-currency figure
(see "`cost` vs `cost_currency_normalized`" below) because its contract
was built to reproduce specific historically-validated thesis numbers.
Both read from the same `_full_analysis()` call underneath - only the
field each one *exposes* as "cost" differs.

**Integrity check**: `redesign_service.redesign()` also verifies
`flow_analysis()`'s cycle time/normalized-cost agree (within 0.05, to
absorb floating-point rounding-order noise between the two independent
graph walks - `metrics.compute_metrics`, used for the RL trainer's reward
signal, and `quantitative_analysis.flow_analysis`, used for the fuller
suite) with `metrics.compute_metrics()`'s numbers for the same graph,
raising `RedesignIntegrityError` if they ever disagree by more. Verified
at exact (0.0) agreement across both sample fixtures and several
post-heuristic graph shapes before this was wired in - see
`test_redesign_service.py::test_metrics_compute_metrics_and_flow_analysis_agree_on_as_is_and_second_process`.

## There is no separate to-be JSON file

`compare(as_is_path)` takes one file. The to-be process is generated by
running the existing `train()` / `replay_sequence()` / `serialize_graph()`
pipeline against the as-is graph - the same thing `POST /redesign` does.
This was confirmed empirically before writing any of this module: running
the existing pipeline on `app/data/asIsProcess.json` reproduces every
validation number below, including the exact wording of the sequence step
description ("Eliminated non-value-adding task 'Start New Program'.").

### Reproducibility (the `seed` parameter)

`app/core/trainer.py`'s Q-learning search was previously unseeded
(`random.Random()`), so two runs against the same as-is process can land on
different heuristic sequences that are equally optimal on cycle time but
not on cost (there are usually several ways to shave the same amount of
time off a small process). Cycle time converged to the same value on every
seed tested; cost did not. `train()` now accepts an optional `seed`, and
`compare()` defaults to `seed=42` so the report is reproducible run to run
- pass `seed=None` to restore the original fully-stochastic behavior (what
`POST /redesign` still uses).

## `cost` vs `cost_currency_normalized` - why the "buggy" number is primary

Every RACI entry's job carries a `currencyType` (PKR/USD/EUR/GBP in the
sample data). This repo already had a fix for the resulting mixed-currency
bug (`app/core/fx.py`, applied at parse time in `parser.build_graph`), and
`KNOWN_LIMITATIONS.md` documents it. The four validation numbers above,
however, are the **pre-fix, uncorrected** totals - summing `hourlyRate`
across currencies as if they were all the same unit. That's what the
`cost` field in the JSON contract holds, for two reasons:

1. It's the number that was already hand-verified independently, so it's
   the one a reader can check.
2. It's what the wider engine's `aggregate()` function and any existing
   dashboards built against the pre-fix numbers expect - changing the
   primary `cost` field's meaning would silently break those without any
   type or shape change to signal it.

The corrected, currency-normalized figure (all rates converted to PKR via
`app/core/fx.py`, same FX table the rest of the engine uses) is reported
alongside it as `cost_currency_normalized` on every `*_flow_analysis`
block, so the delta the bug was causing is visible: as-is normalized cost
is ~4.8x the raw figure for this process, because PKR is a ~278x-smaller
unit than USD/EUR and most of the sample RACI entries are denominated in
USD/EUR. **`cost_currency_normalized` is the number to trust for anything
resembling real money; `cost` exists for continuity with the legacy
figures.** `improvement.cost_percent` is computed from `cost` (raw) to
match the validated 41.22% figure; if you want the improvement percentage
on the corrected basis, compute it from the two `cost_currency_normalized`
values instead.

FX rates: reused `app/core/fx.py`'s existing table verbatim rather than
inventing a new one - it fetches live USD-based rates from
`open.er-api.com` (24h cache) and falls back to a hardcoded snapshot
(`FX_FETCH_TIMEOUT_SECONDS`-bounded) if the live fetch fails:

```
USD: 1.0   PKR: 278.211558   EUR: 0.875764   GBP: 0.747014
```

dated 2026-07-15 in that file's own comment. Override by editing
`FALLBACK_RATES` there (this module doesn't maintain a second copy).

No fixed/overhead cost field exists anywhere in the schema, so every rupee
in `cost`/`cost_currency_normalized` is variable (resource-time-driven)
cost.

## Rework-loop handling

`expected_rework_time` is a flat per-task field, already summed into cycle
time and cost by the ordinary Sequential/Conditional/Parallel composition
with no further adjustment - and that alone reproduces the validated
totals exactly. That means it's already an expected value computed
upstream (outside this engine); applying the Repetition formula
(`CT = CT(single_pass) * 1/(1-p)`) to it again would double-count the loop.

The Repetition pattern is instead applied only when the graph itself
contains an actual back edge: a gateway branch whose target is an ancestor
already on the current path from START. `asIsProcess.json` has none (all
three gateways are `EXCLUSIVE`/`INCLUSIVE` acting as one-shot XOR splits),
so this never fires on the real data - it's implemented and tested purely
for genericity, using two synthetic fixtures in
`test_quantitative_analysis.py` (a 30%-loopback rework cycle, and a
p=1 "never exits" case that raises instead of dividing by zero).

Implementation: `resolve_loops()` is a preprocessing pass that finds each
back edge via DFS (recursion-stack membership = ancestor), computes the
loop body's single-pass value by walking the sub-DAG from the loop's entry
node to the looping gateway (treated as a sink), applies the Repetition
formula, and folds the whole loop into one synthetic node wired to
whatever the gateway's non-looping (exit) branch(es) led to - branch
probabilities on the exit side are renormalized to sum to 1, since the
loop's expected repetition is already fully accounted for by that point.
The ordinary Sequential/Conditional/Parallel walker then runs on the
resulting (guaranteed acyclic) graph unmodified. Nested loops are folded
innermost-first by re-running detection after every fold. **Known
limitation**, in the same spirit as this repo's existing
`KNOWN_LIMITATIONS.md`: deeply overlapping/interleaved loop structures
(not simple nesting) aren't handled and will raise rather than silently
produce a wrong number.

## `sequence`: all 10 heuristics, not just the applied ones

`sequence` lists one entry per heuristic in `app.core.heuristics.HEURISTICS`
(always 10 distinct `heuristic_id` values, in that registry's fixed order)
so a frontend can render a stable checklist of everything the redesign
search considered, not just what it used. A heuristic applied more than
once (e.g. two `extra_resources` calls on different tasks) gets one row
per application, each carrying its 1-based `step_order` in the actual
execution sequence; a heuristic never applied gets a single row with
`applied: false`, `target: null`, `step_order: null`, and a `description`
that distinguishes two different reasons: `"No applicable candidates
found in this process."` (the heuristic's structural preconditions were
never met on this graph) vs. `"Candidates were available on this process
but were not selected by the redesign search."` (it could have fired but
the Q-learning search found something better). That distinction comes
from re-running `all_candidates()` against the baseline graph independent
of what the RL search happened to pick.

## Cycle Time Efficiency

`CTE = value_add_minutes / cycle_time_minutes`. Per the spec, only
`value_classification == "VA"` counts as value-adding; `"BVA"` and `"NVA"`
both count as non-value-adding (BVA is "necessary but not customer-value-
adding" in the strict Ch.7 sense). `improvement.cte_percent_change` is
positive when CTE *increases* (unlike the cost/cycle-time/WIP fields,
where positive means the value went *down* - CTE improving is a rise, and
the sign was chosen so "positive = better" holds for every field in
`improvement` at a glance).

## Resource utilization & Little's Law

- **Required capacity** per job = `reachability_probability(task) x
  arrival_rate_per_week x (process_time + rework_time) x
  (time_allocation_percentage / 100)`, summed over every task the job is
  staffed on. `expected_waiting_time` is deliberately excluded: it's
  elapsed calendar time a task instance sits idle (e.g. waiting on an
  external party), not time the resource is actually occupied, so it
  shouldn't count against that resource's capacity.
- **Reachability probability** is a forward pass over the graph (after
  `resolve_loops`): XOR/inclusive branches inherit a fraction of the
  parent's probability; AND/parallel branches each inherit the parent's
  probability in full, since every parallel branch executes whenever the
  split is reached. This keeps low-probability conditional branches from
  inflating a resource's required capacity.
- **Instance frequency** (`frequency_interval`/`frequency_period`/
  `occurrences`) lives on each *task* record in this schema, but
  describes how often the *process* runs, not the individual task - every
  task in the sample process agrees (`1` per `WEEK`), so
  `process_instance_frequency_per_week()` takes it from the first
  reachable task in topological order and treats it as the process-wide
  arrival rate (lambda). If a process definition ever has genuinely
  disagreeing per-task frequencies, this will silently use whichever task
  is reached first - that's a schema ambiguity, not something this module
  can resolve on its own.
- **Available capacity** per job, per week = `hours_per_day x
  days_per_week x 60 x (1 - capacity_buffer / 100)` (buffer defaults to 0
  when absent, e.g. time already reserved for other duties/PTO).
- **Utilization** = required / available, as a percentage; resources are
  ranked descending, and the top-ranked one is reported as the
  `bottleneck` regardless of whether it exceeds 100% - a `utilization
  > 100` flags a true structural bottleneck (over-allocated, not just the
  single longest task), which is why this ranking can (and, on the sample
  data, does) differ from whichever task has the largest
  `expected_process_time`.
- **WIP** (Little's Law): `WIP = arrival_rate_per_week x
  cycle_time_in_weeks`, converting cycle time from minutes to weeks by
  dividing by `7*24*60` (calendar time, not just working hours - Little's
  Law doesn't care which clock cycle time was measured on, as long as
  arrival rate is expressed on the same one).
